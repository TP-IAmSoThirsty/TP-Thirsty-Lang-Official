## Abstract

Most programming languages treat *authorization* as an application concern bolted
on after the fact: a function runs, and somewhere inside it a check may or may not
happen before it writes a file, opens a socket, or shells out. When code is
increasingly **generated by AI agents**, that model fails closed nowhere ‚Äî a
plausibly-worded function can reach the filesystem, the network, or a subprocess
with no gate in its path.

Thirsty-Lang inverts the default. In **governed** mode, a side effect cannot occur
before an explicit **ALLOW** verdict from a separate policy engine
(T.A.R.L.), and every governed decision emits a tamper-evident **proof**
certificate binding the policy, the context, and the verdict. Missing policy,
missing authority, a failed signature check, or an unavailable gate all resolve to
**DENY**. Governance is the substrate, not a library call.

This paper describes the language, its governance model, the T.A.R.L. policy
engine and proof system, the offensive threat model that scopes its guarantees,
and the engineering state of the 0.5.0 release.

## 1. Motivation and threat setting

Thirsty-Lang targets a specific adversary class: **untrusted code and untrusted
policy**, including AI-generated programs, prompt-injected agents, malicious
plugins/tool adapters, and operators attempting unauthorized actions. The full
adversary catalog (A1‚ÄìA14) and a 50-entry offensive challenge catalog (C001‚ÄìC050)
are maintained in [THREAT_MODEL.md](THREAT_MODEL.md).

The design goal is not "mathematically unbreakable." It is: **no known reproducible
critical/high bypass of the governed runtime remains, and every remaining gap is
recorded with severity and a next step.** The threat catalog is the ledger that
keeps that claim honest.

### Mandatory invariants

These define the substrate. Violating any one downgrades Thirsty-Lang from an
"active resistance runtime" to a "policy library":

1. **No effect before verdict.** A side effect cannot happen before ALLOW.
2. **No policy means DENY.** Missing policy never grants authority.
3. **No authority means DENY.** Missing authority never grants authority.
4. **No proof means no audit claim.** Execution without proof is not governed execution.
5. **No unsigned proof in hardened mode.** Hardened deployments require Ed25519.
6. **No stale proof as authority.** Proofs describe decisions; they are not reusable tokens.
7. **No adapter side doors.** Tool, file, network, subprocess, import, FFI, DB, and model-action paths go through the broker.
8. **No catch-and-continue for governance denials.** Application error handling cannot swallow a denial.
9. **No silent downgrade.** If signing, policy, audit, or identity verification is required and unavailable, fail closed.
10. **No unverifiable readiness claim.** Security claims cite tests or are labeled roadmap.

## 2. Architecture: the Universal Thirsty Family

UTF is a six-tier stack. Each tier adds capability while preserving default-DENY
governance at every gate. The Python reference implementation ships them as one
package (`utf`) with seven console entry points.

| Tier | Name | Role | CLI |
|---|---|---|---|
| 1 | **Thirsty-Lang (Core)** | Water-metaphor language: lexer, parser, tree-walking interpreter, type checker | `thirsty` |
| 2 | **Thirst of Gods** | Async/effect-linked execution surface (`cascade`/`spillage`) | `thirst-of-gods` |
| 3 | **T.A.R.L.** | Thirsty's Active Resistance Language ‚Äî the policy engine + proof system | `tarl`, `tarl-lsp` |
| 4 | **Shadow Thirst** | Mutation governance: prove a candidate computes the same thing as a validated canonical | `shadow-thirst` |
| 5 | **TSCG** | Thirsty Symbolic Constraint Grammar ‚Äî symbolic integrity constraints | `tscg` |
| 6 | **TSCG-B** | TSCG-Binary ‚Äî binary-frame integrity | `tscg-b` |

**Runtime:** Python ‚â• 3.11, one required dependency (`cryptography`, for Ed25519/
HMAC proof signatures). The Z3 SMT layer used for static policy analysis and
symbolic convergence is an **optional** extra (`pip install thirsty-lang[analysis]`);
the package installs and runs without it, and z3-backed features degrade to an
"install the extra" message rather than failing.

## 3. The language (Tier 1)

Thirsty-Lang uses a water metaphor: `drink` declares a binding, `pour` emits,
`sip` reads, `glass` declares a function, `fountain` a class, `reservoir` a list,
`thirsty/hydrated` is if/else, `refill` loops. A module declares its governance
**mode** in its header:

```thirsty
module billing: governed

glass charge(amount: Int) requires amount > 0 ensures result >= 0 {
    return amount
}

drink total = charge(42)
pour total
```

Module modes:

- **`core`** ‚Äî ordinary execution; capability gates are inert.
- **`governed`** ‚Äî every gated capability (I/O, import, sensitive stdlib call)
  must be authorized by a policy verdict; deny-by-default.
- **`strict`** ‚Äî `governed` plus mandatory initialization of declarations.
- **`pure`** ‚Äî no I/O permitted (`pour`/`sip` rejected).

The interpreter supports first-class functions, closures, OOP (fields, methods,
`new`), reservoirs/maps, `cascade` (await-and-yield), `spillage`/`error`/`cleanup`
control flow, pipelines (`|`, `->`), and tail-call optimization at higher
optimization levels. A 16-namespace standard library (`thirst::fs`, `http`, `net`,
`process`, `env`, `sqlite`, `crypto`, `json`, ‚Ä¶) plus 16 global builtins is
available through `import`.

Design-by-contract is built in: `requires` (precondition), `ensures`
(postcondition, with `result` bound), and `invariant` (checked at entry **and**
exit) are executable runtime checks on governed functions, and contract
predicates on methods hold in any mode.

## 4. The governance model (Tier 3 in action)

### 4.1 Capability gates, fail-closed

In governed mode, every side-effecting operation routes through
`_gate_capability(action, target)` before it runs. The gate is **fail-closed**:

- If no policy engine and authority are attached, the capability is **denied**
  with a proof ‚Äî governed mode never implies authority.
- Otherwise the action/target/authority context is evaluated by T.A.R.L.; only an
  explicit **ALLOW** lets the effect proceed. Deny-by-default (no matching rule ‚Üí
  DENY) comes from the engine itself.

Gated capabilities include `pour`/`print` (write stdout), `sip` (read stdin),
`import`, and **every sensitive imported-stdlib call**. The capability/action map
for the standard library is a single source of truth
(`module_system.SENSITIVE_STDLIB_CAPABILITIES`): each sensitive callable carries an
explicit `read` / `write` / `network` / `execute` action, so allowing
`action == "import"` grants no filesystem write, network call, subprocess,
environment mutation, logging, or SQLite side effect (challenges C029‚ÄìC032).

### 4.2 Imports execute under the caller's governance

An imported `.thirsty` module runs under the **caller's** governed runtime ‚Äî
sharing its policy engine and authority and forced into governed mode ‚Äî rather
than in a detached, ungoverned interpreter. Top-level effects in the imported
module are gated at import time, and the function closures it exports remain gated
when later invoked from governed code. A denial during import surfaces as a
`GovernanceViolation` and is **not** swallowed by `spillage` (challenge C035).

### 4.3 The hard floor

`GovernanceViolation` is a control-flow floor: `spillage` and other error handlers
must not catch it, so a governed denial cannot be absorbed by application error
handling (challenge C006). Cross-mode calls (invoking a governed function from
`core`) are rejected statically (E053) and at runtime.

### 4.4 Parser fail-closed

A `governed` module with **any** parse error fails closed: the parser discards all
recovered statements (no executable program survives), and the interpreter refuses
to run it, raising a denial with a proof. Parser error-recovery therefore cannot
smuggle executable statements past a malformed boundary (challenge C036).
Non-governed modules keep ordinary error recovery.

## 5. T.A.R.L. ‚Äî the policy engine and proof system

T.A.R.L. (Thirsty's Active Resistance Language) is a standalone policy layer.

### 5.1 Policy language and evaluation

Policies are ordered rule lists with **first-match-wins** semantics and three
verdicts ‚Äî **ALLOW**, **DENY**, **ESCALATE** (for human/quorum workflows):

```tarl
policy access
when role == "admin" => ALLOW
when action == "read" and source:allowlist => ALLOW
when true => DENY            # explicit default-deny floor
```

The runtime evaluates rules over a context dict, with a safe expression evaluator
(no `eval`/import/state mutation in conditions), an LRU decision cache, and
adaptive rule ordering that never changes first-match semantics. A dynamic
**source registry** binds live data providers to `source:<name>` references.
Unknown identifiers fail safe; evaluator exceptions are observable via `throw_stats`
and treated as non-matching.

### 5.2 Temporal governance

Rules and policies carry time windows (`valid_from`/`valid_until`, durations like
`for: 4h`, `if_unresolved_after ‚Ä¶ revert_to ‚Ä¶`). Temporally-constrained decisions
are never cached, so an expiring ALLOW cannot be replayed from cache.

### 5.3 Composition

Policies compose (`EXTENDS`/`RESTRICTS`) and group into policy sets with set
operators (`UNION`/`INTERSECT`/`MAJORITY`). Composition cycles raise an error
rather than failing open.

### 5.4 Proof-carrying decisions

Every governed boundary decision ‚Äî policy verdict, contract verdict, or
fail-closed denial ‚Äî emits a `TarlProof` certificate binding:

- `policy_hash` ‚Äî SHA-256 of the policy source,
- `context_hash` ‚Äî SHA-256 of the canonical context,
- the matched rule index and condition, the verdict, and a per-rule **trace** up
  to the first match,
- an evaluation timestamp, and an optional signature.

Proofs are **unsigned by default**. Two signing modes exist:

- **HMAC-SHA256** ‚Äî a *symmetric* MAC: detects tampering, but anyone holding the
  shared key (including the verifier) can forge it. Documented as **not**
  non-repudiable.
- **Ed25519** ‚Äî an *asymmetric* signature; the verifier needs only the public key.
  Use this when a proof must attest to the signer.

### 5.5 Independent verification ‚Äî permissive default, strict opt-in

`ProofVerifier` checks signature validity, policy-hash match (when the source is
supplied), and trace consistency ‚Äî with no runtime or policy engine required.
The default is backward-compatible (an unsigned proof verifies). **Hardened
deployments opt in** to strictness:

```python
ProofVerifier(
    require_signature=True,                 # reject unsigned proofs
    allowed_signature_algorithms={"ed25519"},  # reject HMAC / downgrade
    require_policy_source=True,             # require policy binding
)
```

The CLI exposes this as `tarl verify --require-signature --ed25519-only`. Strict
verification rejects unsigned, wrong-key, tampered-field, HMAC-as-Ed25519, and
missing-policy-source proofs (challenge C025).

### 5.6 Static analysis (optional Z3 layer)

With the `analysis` extra, `tarl analyze` proves policy properties via Z3:
**coverage** (contexts that fall through to default-DENY), **shadows** (dead
rules), **conflicts** (overlapping rules with different verdicts), **equiv** (two
policies decide identically for all contexts), and **refines** (one policy never
allows what another would deny). Because z3's bindings share a non-thread-safe
global context, every analysis entry point is serialized through a process-wide
lock with z3 garbage collected inside it ‚Äî so concurrent use (e.g. the LSP running
analysis on a background thread) cannot corrupt the z3 heap.

### 5.7 Audit, explanation, and tooling

T.A.R.L. ships an append-oriented **audit archive** (proofs stored on evaluation,
queryable by verdict/time), an **explainer** (`tarl explain` ‚Äî why a context got a
verdict, including skipped rules), a **test runner** (`.tarl_test` suites), and a
**language server** (`tarl-lsp`, JSON-RPC over stdio or TCP) providing syntax
diagnostics plus z3-backed coverage/shadow hints.

## 6. Mutation governance (Tier 4) and integrity (Tiers 5‚Äì6)

**Shadow Thirst** answers one question about a proposed change: does a *shadow*
block compute the same thing as the *validated canonical* block? Three layers
answer it, cheapest-and-strongest first:

1. **Structural** ‚Äî alpha-renamed AST equality (a sound sufficient condition).
2. **Z3 symbolic** (optional) ‚Äî translate each block's returned value to a Z3 term
   over shared symbolic inputs; `UNSAT` proves equivalence for all inputs, `SAT`
   yields a concrete diverging counterexample.
3. **Execute-and-compare** ‚Äî run both in a sandboxed interpreter over seeded
   inputs and compare; abstains when blocks have observable effects.

A six-analyzer pipeline (plane isolation, determinism, resource estimation, purity,
memory evaporation, canonical convergence) drives a promote/reject verdict.
**TSCG/TSCG-B** provide symbolic-constraint and binary-frame integrity surfaces.

## 7. Build outputs and governance loss

Thirsty-Lang can transpile to JavaScript, LLVM IR (and via the toolchain to
object/exe/asm/JIT), and a Pyodide (WASM) HTML bundle. None of these targets carry
the governed runtime, so a governed module compiled to them **loses policy
enforcement**. The build therefore **refuses** a governance-dropping target for
governed source by default. `thirsty build --allow-governance-loss` is required to
proceed; it warns on stderr and records `build.governance_loss = true` in the
emitted manifest (challenge C034). Governance loss is never silent.

## 8. Security posture: covered vs. deferred

The threat catalog (C001‚ÄìC050) is the source of truth. As of 0.5.0:

Every challenge C001‚ÄìC050 is now **Covered (test-backed)**. Highlights:

- **Fail-closed core** ‚Äî governed I/O/import/print without policy (C001‚ÄìC004);
  non-matching capability denial (C005); denial unswallowable by `spillage`
  (C006); cross-mode guard (C007); evaluator/audit failure ‚Üí DENY (C037‚ÄìC038).
- **Proof integrity & freshness** ‚Äî tamper/wrong-key/no-key/policy-swap detection
  (C009‚ÄìC014); downgrade rejection (C025); context-bound replay, freshness
  windows, policy revocation, single-use replay guard (C023‚ÄìC024).
- **Authority provenance** ‚Äî Ed25519-signed `AuthorityClaim`s; a bare
  `--authority` string grants nothing in hardened mode (C027‚ÄìC028); hardened
  mode also mandates Ed25519-signed proofs.
- **Tamper-evident audit** ‚Äî hash-linked archive whose `verify_chain` detects
  edits, deletions, and reordering (C022, C026, C049).
- **Universal broker** ‚Äî one `CapabilityBroker` mediates FFI/native (C033) and
  agent/MCP tool adapters (C040‚ÄìC041), denied by default.
- **Context & path safety** ‚Äî schema validation rejects missing/type-confused
  context (C045‚ÄìC046); `PathGuard` confines canonical paths against traversal and
  symlink escape (C042).
- **Time & escalation** ‚Äî `TrustedClock` decides temporal windows on verified
  signed time, not the host clock (C043); `QuorumResolver` upgrades ESCALATE only
  on distinct signed approvals (C050); the linter flags broad/ungated ALLOW
  (C039).

**Remaining work is breadth and operations, not a known critical bypass:** the
shipped stdlib adapters use the (equivalent) interpreter gate rather than all
routing through `CapabilityBroker`; durable cross-process replay/audit state and
key provisioning/rotation are left to the embedding; automatic schema derivation
from a policy is future work. See THREAT_MODEL.md ¬ß"Remaining Gaps".

## 10. Getting started

```bash
pip install thirsty-lang                 # core
pip install thirsty-lang[analysis]       # + Z3 static analysis & symbolic convergence

thirsty run program.thirsty --thirst-level governed \
    --authority admin --policy access.tarl

tarl eval access.tarl -c '{"role":"admin"}'
tarl analyze coverage access.tarl
tarl verify proof.json --require-signature --ed25519-only --ed25519-key id:HEX
```

A denied governed call prints `governance denied: <fn>: <reason>` with the proof
verdict and policy hash, and exits non-zero.

## 11. Conclusion

Thirsty-Lang makes authorization a property of the runtime rather than a
convention of the code: in governed mode no effect precedes an explicit ALLOW,
every decision carries a verifiable proof, and every "secure" claim is pinned to a
test or labeled as roadmap. The 0.5.0 release closes the import, build, parser, and
proof-downgrade bypasses in the offensive catalog and records the rest with
severity and next steps. The remaining work ‚Äî authority provenance, tamper-evident
audit, replay/downgrade rejection, and universal adapter brokering ‚Äî is what
stands between the current alpha and a deployable governance substrate.
